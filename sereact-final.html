PROJECT OVERVIEW

Autonomous bin picking through intelligent cell management

This project involved planning and programming the control system for a Robotic Cell designed for bin picking applications in warehouse environments. A robotic cell is a secured environment where a robotic arm safely operates, equipped with safety features including doors for human access and stack-lights to communicate system status. The system interfaces with warehouse operations through a Human-Machine Interface (HMI), enabling seamless interaction between workers and the automated picking system.

The challenge was to create a robust, real-time control system that manages multiple subsystems including API communication with the Warehouse Management System (WMS), barcode scanning, safety monitoring (door handles and emergency buttons), and visual status indication through stack-lights. The entire system was built using ROS 2 (Robot Operating System 2), ensuring modularity, reliability, and industry-standard communication protocols.

Working independently over three months, I developed six major components that work together to create a complete warehouse automation solution: an API server for WMS communication, ROS 2 nodes for hardware integration (scanner, door, emergency button, stack-light), and a comprehensive HMI for real-time monitoring and control. This project also provided valuable hands-on experience with Docker containerization, a critical skill for modern robotics deployment.

BACKGROUND

Understanding the robotic picking cell

In modern warehouse operations, robotic cells represent a critical intersection of automation, safety, and efficiency. A robotic picking cell is a secured environment where industrial robotic arms perform bin picking operations‚Äîselecting items from containers and placing them in designated locations. These cells are designed with multiple safety and communication features to ensure smooth integration with human workers and existing warehouse management systems.

Key Components of a Robotic Cell

  • Robotic Arm: The core automation element that performs physical picking operations
  • Safety Enclosure: Physical barriers with access doors that protect workers from the robot's operational space
  • Barcode Scanners: Vision systems that identify items during the picking process
  • Stack-Light: Visual indicator showing system status (operational, paused, emergency)
  • Emergency Stop Button: Critical safety feature for immediate robot shutdown
  • Door Sensors: Monitor access points to ensure worker safety
  • HMI (Human-Machine Interface): Touchscreen display for monitoring and control

The Warehouse Management System (WMS) orchestrates picking operations by sending requests to the robotic cell, specifying which items to pick and their quantities. The cell must respond with accurate status updates and completion confirmations, creating a continuous feedback loop that keeps warehouse operations running efficiently.

DESIGN CHALLENGE

Building a complete control system from scratch

The primary challenge was to design and implement a complete control system that could manage all aspects of the robotic cell while maintaining reliability, safety, and real-time responsiveness. Unlike simulation environments, this system would interface with real hardware and production systems, requiring robust error handling and fail-safe mechanisms.

Technical Challenges

  • Real-time Integration: The system needed to process API requests, monitor hardware states, and update the HMI simultaneously without delays
  • Safety-Critical Operations: Emergency stops and door sensors required immediate response‚Äîany delay could compromise worker safety
  • API Design: Creating both synchronous and asynchronous communication patterns between the WMS and robotic cell
  • State Management: Coordinating multiple subsystems (door state, emergency button, scanner, stack-light) to determine overall system status
  • Mock Hardware Development: Since physical hardware wasn't available, I had to create realistic simulations that accurately represented sensor behaviors

The solution required implementing ROS 2's publisher-subscriber architecture, HTTP server capabilities, and a real-time HMI‚Äîall working together in a cohesive system that could be deployed in a production warehouse environment.

TASK 1: API COMMUNICATION

WMS integration through REST API

The Warehouse Management System communicates with the robotic cell through API calls, sending picking requests and receiving completion confirmations. I implemented a complete server-side solution that handles these requests according to Sereact's specifications.

Pick Request Endpoint

When the WMS needs items picked, it sends a POST request to the cell:

HTTP POST http://<ip-address>:8080/pick
JSON Request Body:
{
  "pickId": 123,
  "quantity": 4
}

The server receives this request and queues the picking operation. The system can be implemented either synchronously (blocking until completion) or asynchronously (immediate response with later confirmation).

Pick Confirmation Endpoint

Once the robot completes the picking operation, the cell sends a confirmation back to the WMS:

HTTP POST http://<ip-address>:8081/confirmPick
JSON Request Body:
{
  "pickId": 123,
  "pickSuccessful": true,
  "errorMessage": null,
  "itemBarcode": 123
}

Implementation Highlights

  • Built HTTP server using Python's Flask framework
  • Implemented request validation and error handling
  • Created mock picking logic that simulates robot operations
  • Integrated barcode scanner data into pick confirmations
  • Designed response format that includes success status and error messages

The itemBarcode field in the confirmation comes from the barcode scanner node, ensuring that the correct item was picked. If picking fails, the errorMessage provides diagnostic information to help warehouse operators troubleshoot issues.

TASK 2: BARCODE SCANNING

Scanner ROS 2 node implementation

Barcode scanning is essential for verifying that the robot picks the correct items. In production environments, multiple barcode scanners read item numbers during the picking process, making this data available to the control system for validation.

System Requirements

I implemented a ROS 2 node that continuously publishes randomly generated 5-digit barcodes to simulate scanner operation. Additionally, I created a service that returns the most recently scanned barcode without requiring any input parameters.

Architecture

  • Publisher: Continuously generates and publishes 5-digit barcodes at regular intervals
  • Service: Provides on-demand access to the latest scanned barcode
  • Topic: /barcode_scan for publishing barcode data
  • Message Type: String containing the barcode number

In the actual implementation, the publisher would receive data from physical barcode scanners connected via USB or network. The service allows other nodes (particularly the API handler) to retrieve the current barcode when confirming pick operations.

Use in Pick Confirmation

When a pick operation completes, the API handler calls the scanner service to retrieve the barcode, which is then included in the confirmation message sent back to the WMS. This creates an audit trail confirming that the correct item was picked.

TASK 3: SAFETY MONITORING

Door handle state management

Safety is paramount in robotic cells. All cells include at least one access door that allows workers to enter for maintenance or manual operations. The control system must constantly monitor door state to prevent robot operation when doors are open.

Safety Logic

The fundamental safety rule is: When the door is open, the robot must not move. All pick requests received while the door is open must be responded to with pickSuccessful: false.

ROS 2 Node Implementation

  • Publisher: Continuously publishes door state (boolean value)
  • True: Door is closed (safe to operate)
  • False: Door is open (operation prohibited)
  • Service: Allows toggling door state for testing purposes

Mock Operation

Since physical door sensors weren't available, I implemented a service that can manually change the door state. This allows testing the system's response to door events:

  • Calling the service toggles between open (false) and closed (true)
  • Alternatively, killing and restarting the node changes the state
  • The API handler subscribes to door state and blocks operations when necessary

In production, this node would interface with actual door sensors (magnetic switches, optical sensors, or mechanical limit switches) to detect the real door position.

TASK 4: EMERGENCY STOP SYSTEM

Emergency button monitoring and control

Emergency stop buttons (e-buttons) are critical safety devices that allow workers to immediately halt all robotic motion. Once pressed, the robot remains stopped until the button is explicitly released by an operator.

Safety Protocol

When the emergency button is pressed:

  • The robot stops all motion immediately
  • The cell enters emergency mode and cannot return to operational mode
  • All pick requests are rejected with pickSuccessful: false
  • The system remains in this state until the button is released

ROS 2 Node Design

  • Publisher: Continuously publishes e-button state (boolean)
  • True: Button is pressed (emergency state)
  • False: Button is not pressed (normal operation possible)
  • Services: Two services for testing
    • Press Service: Sets state to true (simulates button press)
    • Release Service: Sets state to false (simulates button release)

Integration with Cell Control

The API handler subscribes to the emergency button state. When an emergency is detected, the handler immediately rejects all incoming pick requests and updates the HMI to show emergency status. This ensures that warehouse operators are immediately aware of safety issues.

TASK 5: VISUAL STATUS INDICATION

Stack-light control system

Stack-lights are industrial visual indicators that use colored lights to communicate system status at a glance. Workers can quickly assess the cell's operational state from across the warehouse floor without needing to check the HMI.

Status Indicators

The stack-light displays three states based on the overall cell condition:

  • 0 (Green): Operational - Door closed, no emergency, system receiving requests
  • 1 (Yellow): Paused - Door open, robot cannot move but no emergency
  • -1 (Red): Emergency - E-button pressed, immediate attention required

State Determination Logic

The stack-light node subscribes to both the door handle and emergency button topics. It applies the following logic to determine which color to display:

  • If emergency button is pressed ‚Üí Red (-1)
  • Else if door is open ‚Üí Yellow (1)
  • Else (door closed, no emergency) ‚Üí Green (0)

ROS 2 Implementation

  • Subscribers: Listens to door state and emergency button topics
  • Publisher: Publishes current stack-light value (0, 1, or -1)
  • Real-time Updates: Changes immediately when door or emergency state changes

This creates a dynamic visual feedback system that helps warehouse workers understand cell status without detailed technical knowledge. Green means everything is working, yellow means wait for the door to close, and red means immediate intervention is needed.

TASK 6: HUMAN-MACHINE INTERFACE

Real-time monitoring dashboard

The HMI serves as the primary interface between warehouse workers and the robotic cell. It provides real-time visibility into system status and displays critical information about ongoing operations.

Required Information Display

  • Request Information: Current pick request details (pickId, quantity)
  • Response Information: Last pick confirmation sent to WMS (success/failure, barcode)
  • Emergency Button State: Current e-button status (pressed/released)
  • Door Handle State: Current door position (open/closed)
  • Stack-Light State: Visual indicator with color representation
    • Green circle/square for operational (0)
    • Yellow circle/square for paused (1)
    • Red circle/square for emergency (-1)

Technology Stack

I implemented the HMI using Flask web framework for the backend and vanilla JavaScript for the frontend:

  • Flask Server: Python web framework serving the dashboard on port 5000
  • HTML/CSS: Clean, professional interface with dark theme optimized for warehouse environments
  • JavaScript: Real-time data updates using fetch API
  • ROS 2 Integration: Subscribers to all system topics for live monitoring
  • Responsive Design: Works on various screen sizes from tablets to large monitors

Real-Time Updates

The HMI subscribes to all relevant ROS 2 topics and updates the display immediately when changes occur. This ensures that operators always see current system status without needing to refresh or manually check. The interface provides immediate feedback when pick requests arrive, when operations complete, or when safety systems activate.

SYSTEM ARCHITECTURE

Integrated control system design

The complete system integrates six major components that communicate through ROS 2's publish-subscribe architecture and service calls. This modular design ensures that each component can be developed, tested, and maintained independently while contributing to the overall system functionality.

Component Interaction Flow

  • WMS ‚Üí API Handler: Pick requests arrive via HTTP POST
  • API Handler ‚Üí Scanner: Service call to retrieve current barcode
  • Door/Emergency ‚Üí Stack-Light: State changes update visual indicator
  • All Nodes ‚Üí HMI: Real-time data streams for monitoring
  • API Handler ‚Üí WMS: Pick confirmation via HTTP POST

Safety Integration

The system implements a layered safety approach where multiple components contribute to operational decisions:

  • Door sensor provides the first level of safety (physical barrier)
  • Emergency button provides immediate shutdown capability
  • Stack-light provides visual status for situational awareness
  • API handler checks all safety conditions before accepting requests
  • HMI displays all safety states for operator awareness

Deployment & Containerization

The system includes Docker containerization for simplified deployment:

  • Single Command Deployment: docker-compose up starts all six components
  • Port Management: Automatically exposes HMI (5000), WMS receive (8080), and confirmation (8081) ports
  • Host Networking: Ensures ROS 2 DDS communication works seamlessly
  • Environment Consistency: Same behavior across development and production machines
  • Easy Scaling: Can deploy multiple cells with different port configurations

TECHNICAL IMPLEMENTATION

Development approach and technologies

ROS 2 Humble

I chose ROS 2 Humble as the foundation for this project because it's the recommended distribution for production systems (ROS Noetic is reaching end-of-life). ROS 2 provides several advantages over ROS 1:

  • Better real-time performance through DDS middleware
  • Improved security features for industrial applications
  • Native support for multiple platforms
  • More reliable service call mechanisms

Python Implementation

All ROS 2 nodes were implemented in Python for rapid development and easy integration with web frameworks. Python's extensive library ecosystem provided the perfect tools for this project:

  • Flask Web Framework: Used for both the WMS API server (ports 8080/8081) and the HMI dashboard (port 5000)
  • Requests Library: HTTP client for sending pick confirmations back to WMS
  • Threading: Background task processing for concurrent pick operations
  • Random Module: Simulating pick success rates and barcode generation
  • ROS 2 Python Client (rclpy): All node implementations and pub/sub communication

Docker Containerization

During this project, I learned and implemented Docker containerization to ensure consistent deployment across different environments. This was a significant learning experience that added production-readiness to the system:

  • Dockerfile Creation: Built custom Docker image with ROS 2 Humble, Python dependencies, and system packages
  • Docker Compose: Orchestrated multi-port networking (5000 for HMI, 8080/8081 for API endpoints)
  • Network Configuration: Implemented host networking mode for seamless ROS 2 communication
  • Port Mapping: Exposed multiple services through container port forwarding
  • Environment Isolation: Ensured the system runs identically on development and production machines

The Docker implementation allows the entire robotic cell control system to be deployed with a single command, eliminating "it works on my machine" problems and simplifying warehouse deployment.

Version Control & Documentation

The project followed software engineering best practices:

  • Git version control for tracking all changes
  • Comprehensive README with setup and installation instructions
  • Documentation of dependencies and required packages
  • Clear instructions for running each component
  • Code comments explaining complex logic
  • Docker deployment documentation for production environments

Testing Approach

Since physical hardware wasn't available, I created mock implementations that accurately simulate real-world behavior:

  • Random barcode generation simulating scanner input
  • Service-based state toggling for door and emergency button
  • Simulated picking delays to mimic robot operation time
  • Error injection capabilities for testing failure handling
